    👥 역할별 가이드 #

    🔧 개발자 (Developer) 가이드 #

    대시보드 개요 #

    개발자 대시보드는 단위 테스트 관리에 최적화되어 있습니다.

    주요 기능 #

    1. 내 테스트 케이스 확인 #

    • 대시보드 중앙에서 내가 작성한 테스트 목록 확인
    • 통과/실패 상태를 한눈에 파악
    • 클릭하여 상세 정보 조회

    2. 빠른 테스트 생성 #

    세 가지 방법으로 테스트를 생성할 수 있습니다:

    A) AI 자동 생성 (추천!) # 
    1. "AI 자동 생성" 버튼 클릭
2. 소스코드 파일 업로드 (HTML, JS, CSS, Vue, React 등)
3. 프레임워크 정보 입력 (선택사항)
   - 프레임워크: React, Vue, Angular 등
   - 스타일 라이브러리: Bootstrap, Tailwind 등
   - 프로젝트 설명: 주요 기능 간단히 설명
4. "AI 자동 생성" 버튼 클릭
5. 생성된 테스트 시나리오 검토
6. 필요시 수정 후 "테스트 실행" 클릭

    AI가 자동으로 생성하는 것:

    • 정상 시나리오 테스트
    • 예외 상황 테스트 (빈 필드, 잘못된 입력 등)
    • 에지 케이스 (연속 실패, 최대값 초과 등)
    • 모든 기능의 경우의 수
    B) 수동 작성 # 
    1. "수동 작성" 버튼 클릭
2. "단계 추가" 버튼으로 테스트 케이스 추가
3. 각 테스트 케이스 정보 입력:
   - 테스트 이름: 예) "로그인 버튼 클릭"
   - 실행 동작: 예) "이메일 입력란에 'test@example.com' 입력"
   - 예상 결과: 예) "대시보드 페이지로 이동"
   - 선택자: 예) "#login-button"
4. 추가 케이스가 필요하면 "단계 추가" 반복
5. "테스트 실행" 클릭
    C) 템플릿 사용 # 
    1. "템플릿 사용" 버튼 클릭
2. 사전 정의된 템플릿 선택
3. 프로젝트에 맞게 수정
4. "테스트 실행" 클릭

    3. 테스트 케이스 관리 #

    복제하기:

    • 테스트 케이스 우측의 복사 아이콘 클릭
    • 기존 테스트를 기반으로 새 테스트 생성

    수정하기:

    • 테스트 케이스 카드에서 “수정” 버튼 클릭
    • 내용 변경 후 저장

    삭제하기:

    • 테스트 케이스 카드에서 “삭제” 버튼 클릭
    • 확인 메시지 후 삭제

    4. 테스트 실행 및 결과 확인 # 

    1. 테스트 케이스 카드에서 "실행" 버튼 클릭
2. 실행 중 상태 확인 (스피너 표시)
3. 결과 모달 자동 표시:
   - 통과/실패 개수
   - 실행 시간
   - 각 테스트 케이스별 상세 결과
   - Playwright 실행 화면 (향후)
4. 리포트 저장 또는 확인

    개발자 워크플로우 예시 #

    시나리오: 로그인 기능 테스트 # 

    1. 역할 선택: "개발자" 선택
2. "새 테스트 생성" 클릭
3. 소스코드 업로드:
   - login.html
   - login.js
   - login.css
4. 프레임워크: "React" 선택
5. "AI 자동 생성" 클릭
6. 생성된 4개 시나리오 확인:
   ✓ 정상 로그인
   ✓ 잘못된 비밀번호
   ✓ 빈 필드 검증
   ✓ 3회 이상 실패
7. "테스트 실행" 클릭
8. 결과: 3/4 통과, 1개 실패
9. 실패한 테스트 수정
10. 재실행

    📊 PM/QA 가이드 #

    대시보드 개요 #

    PM/QA 대시보드는 통합 테스트 관리 및 품질 모니터링에 최적화되어 있습니다.

    주요 기능 #

    1. 전체 프로젝트 현황 파악 #

    대시보드 상단에서 다음 정보를 확인:

    • 전체 테스트 수
    • 통과율 (목표: 90% 이상)
    • 처리 필요한 실패 테스트 수
    • 활성 개발자 수

    2. 프로젝트별 상태 모니터링 #

    각 프로젝트 카드에서 확인:

    • 프로젝트 이름 및 상태 (진행중/주의/완료)
    • 통과율 진행바
    • 통과/전체 테스트 비율
    • 담당 개발자 수
    • 마지막 테스트 실행 시간

    색상 코드:

    • 🟢 초록색 (90% 이상): 양호
    • 🟠 주황색 (70-90%): 주의
    • 🔴 빨간색 (70% 미만): 긴급

    3. 개발자별 활동 추적 #

    개발자 활동 섹션에서:

    • 각 개발자의 테스트 통과/실패 개수
    • 활동 순위
    • 품질 지표

    4. 처리 필요 항목 관리 #

    이슈 목록에서:

    • 우선순위별 필터링 (높음/중간/낮음)
    • 연속 실패 테스트 확인
    • “확인” 버튼으로 이슈 처리
    • 개발자에게 피드백 전달

    5. 주간 트렌드 분석 #

    차트에서 확인:

    • 일별 테스트 실행 추이
    • 통과/실패 비율 변화
    • 품질 개선 추세

    PM/QA 워크플로우 예시 #

    시나리오: 주간 품질 리포트 작성 # 

    1. 역할 선택: "PM/QA" 선택
2. 대시보드에서 전체 통계 확인
3. 프로젝트별 상태 검토:
   - E-Commerce: 89% (양호)
   - Admin Dashboard: 95% (우수)
   - Mobile Web: 75% (주의) ← 집중 검토
4. Mobile Web 프로젝트 클릭
5. 실패한 테스트 목록 확인
6. 처리 필요 항목 확인:
   - Payment Test: 높음 (즉시 처리)
   - Search Test: 중간 (금주 내 처리)
7. 개발자별 활동 확인
8. "리포트 생성" 클릭
9. PDF/Excel로 내보내기

    일일 체크리스트 # 

    ☐ 전체 통과율 확인 (목표: 87% 이상)
☐ 신규 실패 테스트 확인
☐ 높음 우선순위 이슈 처리
☐ 개발자 피드백 제공
☐ 주간 트렌드 분석

    ⚙️ 시스템 관리자 (Admin) 가이드 #

    대시보드 개요 #

    관리자 대시보드는 시스템 운영 및 환경 관리에 최적화되어 있습니다.

    주요 기능 #

    1. 시스템 상태 모니터링 #

    실시간으로 다음 항목 확인:

    • 시스템 가동률
    • CPU/메모리/디스크 사용률
    • 등록된 사용자 수
    • Playwright 버전

    알림 기준:

    • 🟢 정상: 가동률 99% 이상
    • 🟡 주의: 가동률 95-99%
    • 🔴 위험: 가동률 95% 미만

    2. 테스트 환경 설정 #

    환경 추가/수정 # 
    1. "설정" 탭 클릭
2. "테스트 환경" 선택
3. 환경 유형 선택:
   - 개발 환경 (Development)
   - 스테이징 환경 (Staging)
   - 프로덕션 환경 (Production)
4. 테스트 URL 입력
5. 배포 정보 입력
6. "저장" 클릭
    환경별 설정 예시 # 
    개발 환경:
- URL: https://dev.example.com
- 용도: 개발자 단위 테스트
- 활성: ✓

스테이징 환경:
- URL: https://staging.example.com
- 용도: QA 통합 테스트
- 활성: ✓

프로덕션 환경:
- URL: https://example.com
- 용도: 최종 검증 (신중히 사용)
- 활성: ✗ (필요시에만 활성화)

    3. 사용자 관리 #

    사용자 추가 # 
    1. "사용자 추가" 버튼 클릭
2. 정보 입력:
   - 이름
   - 이메일
   - 역할 (개발자/PM/관리자)
3. 초기 비밀번호 설정
4. "저장" 클릭
    사용자 수정 # 
    1. 사용자 목록에서 편집 아이콘 클릭
2. 역할 변경 또는 상태 변경
3. "저장" 클릭
    권한 관리 #
    • 개발자: 자신의 테스트만 수정
    • PM/QA: 모든 테스트 조회, 승인/반려
    • 관리자: 모든 기능 접근

    4. Playwright 설정 # 

    1. "설정" → "Playwright 설정" 클릭
2. 브라우저 선택:
   ☑ Chromium
   ☑ Firefox
   ☐ WebKit
3. 헤드리스 모드 설정
4. 타임아웃 설정 (기본: 30초)
5. 스크린샷 옵션 설정
6. "저장" 클릭

    5. 시스템 로그 분석 #

    로그 유형:

    • 🔵 INFO: 일반 정보
    • 🟢 SUCCESS: 성공 작업
    • 🟡 WARNING: 경고
    • 🔴 ERROR: 오류

    필터링 및 검색:

    1. 날짜 범위 선택
2. 로그 레벨 선택
3. 키워드 검색
4. "적용" 클릭

    6. 리소스 모니터링 #

    실시간 모니터링:

    • CPU 사용률: 정상 범위 0-70%
    • 메모리 사용률: 정상 범위 0-80%
    • 디스크 I/O: 정상 범위 0-50%

    이상 징후 시 조치:

    1. 알림 확인
    2. 로그에서 원인 파악
    3. 필요시 서비스 재시작
    4. 리소스 증설 검토

    관리자 일일 체크리스트 # 

    ☐ 시스템 가동률 확인
☐ 리소스 사용률 점검
☐ 에러 로그 검토
☐ 사용자 활동 확인
☐ 백업 상태 확인
☐ Playwright 업데이트 확인

    💡 고급 활용 팁 #

    1. 효율적인 테스트 작성 # 

    좋은 예:
✓ 명확한 테스트 이름: "로그인 - 정상 케이스"
✓ 구체적인 예상 결과: "대시보드 페이지 URL이 '/dashboard'로 변경됨"
✓ 정확한 선택자: "#login-button" (ID 사용)

나쁜 예:
✗ 모호한 이름: "테스트 1"
✗ 불명확한 결과: "제대로 작동함"
✗ 부정확한 선택자: "button" (너무 광범위)

    2. AI 생성 최적화 #

    소스코드 업로드 시:

    • ✅ 관련된 모든 파일 업로드
    • ✅ 프레임워크 정보 정확히 입력
    • ✅ 프로젝트 설명 상세히 작성
    • ✅ 주요 기능 명시

    3. 협업 베스트 프랙티스 #

    • 개발자: 테스트 작성 후 PM/QA에게 리뷰 요청
    • PM/QA: 정기적으로 품질 리포트 공유
    • 관리자: 주간 시스템 점검 리포트 작성

    4. 트러블슈팅 #

    테스트가 실패하는 경우 # 

    1. 선택자 확인: 요소가 존재하는지 검증
2. 타이밍 이슈: 로딩 대기 시간 추가
3. 환경 확인: 올바른 URL 설정 확인
4. 로그 확인: 에러 메시지 분석

    AI 생성이 정확하지 않은 경우 # 

    1. 더 많은 소스코드 업로드
2. 프로젝트 설명 상세화
3. 생성 후 수동으로 수정
4. 템플릿 조합 활용

    🎓 튜토리얼: 처음부터 끝까지 #

    신규 프로젝트 테스트 설정 (30분) #

    1단계: 환경 설정 (관리자, 5분) # 

    1. 관리자 계정으로 로그인
2. 설정 → 테스트 환경
3. 개발 환경 URL 입력
4. Playwright 설정 확인

    2단계: 사용자 추가 (관리자, 5분) # 

    1. 사용자 관리
2. 개발자 3명 추가
3. PM 1명 추가
4. 이메일로 초대장 발송

    3단계: 첫 테스트 작성 (개발자, 10분) # 

    1. 개발자 계정으로 로그인
2. 로그인 페이지 소스코드 업로드
3. AI 자동 생성
4. 4개 시나리오 확인 및 수정
5. 테스트 실행
6. 결과 확인

    4단계: 리뷰 및 승인 (PM, 10분) # 

    1. PM 계정으로 로그인
2. 대시보드에서 신규 테스트 확인
3. 테스트 결과 검토
4. 필요시 피드백 제공
5. 승인 처리

    📞 지원 및 문의 #

    자주 묻는 질문 (FAQ) #

    Q: 소스코드를 업로드하면 보안은 어떻게 되나요? A: 현재 버전은 로컬에서만 실행되므로 외부로 전송되지 않습니다. 향후 서버 버전에서는 암호화 전송을 지원할 예정입니다.

    Q: 몇 개의 테스트 케이스까지 생성 가능한가요? A: 제한 없습니다. 다만, 효율적인 관리를 위해 프로젝트당 50개 이하를 권장합니다.

    Q: Playwright가 설치되어 있어야 하나요? A: 현재 버전은 시뮬레이션이므로 불필요합니다. 실제 통합 버전에서는 필요합니다.
