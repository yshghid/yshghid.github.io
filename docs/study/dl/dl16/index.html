                  BERT 뉴스 분류 #2 BERT와 파인튜닝 #

                  #2026-03-01

                  #1 문제 정의

                  AG News 데이터셋은 뉴스 기사 12만 건으로 이루어져 있다. 네 가지 카테고리(세계, 스포츠, 경제, 과학/기술)에 각각 3만 건씩 균등하게 배분되어 있다.

                  데이터: AG News (뉴스 기사 데이터셋)
  - 학습: 120,000건 (클래스별 30,000건 균등)
  - 테스트: 7,600건

라벨:
  0 → World (세계)
  1 → Sports (스포츠)
  2 → Business (경제)
  3 → Sci/Tech (과학/기술)

문제 유형: 4클래스 분류

                  당뇨병성 망막증 데이터는 등급 0이 25,000장이고 등급 3이 800장이어서 극심한 불균형이 있었다. 그때는 클래스 가중치를 써서 희귀한 클래스의 오분류에 더 큰 벌점을 줘야 했다. 여기서는 네 카테고리가 동일한 수이므로 그런 보정이 필요 없다. 모델이 “전부 스포츠라고 찍기"같은 편법으로 높은 정확도를 얻을 수 없다. 각 카테고리를 진짜로 구분하는 법을 배워야만 성적이 오른다.

                  테스트 세트는 7,600건으로, 학습에 전혀 사용되지 않은 데이터다. 최종 성능을 이 테스트 세트에서 측정한다.

                  #

                  #2 파인튜닝

                  BERT는 위키피디아와 BookCorpus라는 방대한 텍스트(수십억 단어)로 사전 학습된 모델이다. 이 사전 학습 과정에서 BERT는 영어의 문법, 단어 간의 의미적 관계, 문장 구조, 문맥에 따른 단어 의미 변화 같은 언어의 기본기를 익혔다. “bank” 옆에 “river"가 오면 강둑이고 “money"가 오면 은행이라는 걸 이미 안다.

                  하지만 BERT는 뉴스를 분류하는 법은 모른다. 영어를 이해하는 능력과 뉴스 카테고리를 판단하는 능력은 다른 것이다. 영어를 유창하게 하는 사람이라도 “이 기사가 경제 기사인지 과학 기사인지 분류하세요"라는 구체적인 업무는 따로 배워야 한다.파인튜닝은 이 “따로 배우는” 과정이다. 이미 언어를 이해하는 BERT의 가중치를 출발점으로 삼고, 뉴스 분류 데이터로 조금만 더 학습시키는 것이다. 처음부터 학습하는 것과는 차원이 다르다. 영어를 전혀 모르는 상태에서 뉴스 분류를 배우려면 엄청난 데이터와 시간이 필요하지만, 이미 영어를 아는 상태에서는 12만 건의 예시만으로도 충분히 학습할 수 있다.

                  사전학습 BERT: 영어 이해 능력은 있지만 뉴스 분류는 모름
                    ↓ 파인튜닝 (AG News 데이터로 학습)
파인튜닝 BERT: 영어 이해 + 뉴스 카테고리 분류 가능

                  #

                  #3 모델 흐름

                  model = AutoModelForSequenceClassification.from_pretrained(
    "bert-base-uncased", num_labels=4
)

                  먼저 사전학습된 BERT의 가중치 약 1억 1천만 개를 불러온다. 이 가중치들은 수십억 단어를 학습하면서 다듬어진 것이다.

                  그 다음 BERT 위에 분류 레이어 하나를 추가한다. Dense(768 → 4)인데, BERT의 [CLS] 토큰 출력 768차원을 받아서 4개 카테고리에 대한 점수를 내놓는다. 이 분류 레이어만 랜덤하게 초기화되어 있고, 나머지 BERT 부분은 이미 좋은 값을 가지고 있다.

                  입력 텍스트
    ↓ 토크나이저
[input_ids, attention_mask]  shape: (batch, 512)
    ↓ BERT Embedding Layer
토큰 벡터  shape: (batch, 512, 768)
    ↓ Transformer × 12 (Self-Attention + FFN)
문맥화된 벡터  shape: (batch, 512, 768)
    ↓ [CLS] 토큰만 추출
문장 벡터  shape: (batch, 768)
    ↓ Dropout(0.1)
    ↓ Dense(768 → 4)
logits  shape: (batch, 4)
    ↓ softmax
확률  shape: (batch, 4)   예: [0.02, 0.05, 0.91, 0.02] → Business

                  “Wall St. Bears Claw Back Into the Black"이라는 텍스트가 들어오면, 토크나이저가 이걸 [CLS], wall, st, ., bears, claw, back, into, the, black, [SEP]이라는 토큰 시퀀스로 바꾸고, 각 토큰의 사전 번호(input_ids)와 attention_mask를 만든다. 배치 안의 모든 문장이 512 길이로 패딩된다.

                  이 숫자 배열이 BERT의 임베딩 레이어를 통과하면 각 토큰이 768차원 벡터로 변환된다. 단순한 사전 번호가 의미를 담은 벡터가 되는 것이다. 이 시점에서 데이터의 shape은 (배치 크기, 512, 768)이다. 배치 안의 각 문장이 512개 토큰으로 이루어져 있고, 각 토큰이 768차원 벡터로 표현된 상태다.

                  12개의 Transformer 인코더 레이어를 통과하면서 각 토큰의 벡터가 Self-Attention을 통해 다른 토큰들의 정보를 흡수한다. 12겹을 다 통과하면 각 토큰의 768차원 벡터에 문장 전체의 문맥 정보가 녹아들어 있다. shape은 여전히 (배치 크기, 512, 768)이지만, 각 벡터의 내용이 완전히 달라져 있다. 처음의 벡터는 개별 단어의 의미만 담고 있었지만, 이제는 문장 전체의 맥락이 반영된 벡터다.

                  여기서 [CLS] 토큰의 벡터만 뽑아낸다. shape이 (배치 크기, 768)로 줄어든다. 이 768차원 벡터가 문장 전체의 의미를 대표한다. 12겹의 Self-Attention을 거치면서 문장의 모든 단어 정보가 이 위치로 모였기 때문이다. Dropout(0.1)이 이 벡터의 10%를 랜덤하게 꺼서 과적합을 방지한다. 이전 챕터에서 봤던 것과 같은 원리다. 그 다음 Dense(768 → 4)가 768차원을 4개의 숫자(logit)로 압축하고, Softmax가 이걸 확률로 변환한다. [0.02, 0.05, 0.91, 0.02] 같은 결과가 나오면 모델은 “이 기사는 91% 확률로 경제 기사다"라고 판단하는 것이다.

                  이 구조는 당뇨병성 망막증 모델과 놀라울 정도로 유사하다. 거기서는 ResNet이 이미지를 처리해서 1024차원 벡터를 만들고 Dense(5) + Softmax로 5개 등급을 분류했다. 여기서는 BERT가 텍스트를 처리해서 768차원 벡터를 만들고 Dense(4) + Softmax로 4개 카테고리를 분류한다. “입력을 깊은 신경망으로 처리해서 고차원 특징 벡터를 만들고, 그걸 Dense + Softmax로 분류한다"는 패턴이 반복되는 것이다.

                  #

                  #4 학습 설정

                  training_args = TrainingArguments(
    num_train_epochs=1,            # 에포크 수
    per_device_train_batch_size=8, # 배치 크기
    learning_rate=2e-5,            # 학습률
    weight_decay=0.01,             # L2 정규화
    evaluation_strategy="epoch",   # 에포크마다 평가
    load_best_model_at_end=True,   # F1 기준 최고 모델 유지
    metric_for_best_model="f1",
    report_to="mlflow",            # MLflow 자동 기록
)

                  학습 설정에서 가장 눈에 띄는 것은 learning_rate=2e-5, 즉 0.00002라는 극도로 작은 학습률이다. 당뇨병성 망막증 모델에서 1e-5를 썼는데 그것보다도 같은 수준이다.

                  이유는 파인튜닝의 본질에 있다. BERT의 1억 1천만 개 가중치는 수십억 단어를 학습하면서 절묘하게 조율된 값들이다. 이 가중치들이 영어를 이해하는 능력을 담고 있다. 학습률이 크면 한 번의 업데이트로 이 가중치들이 크게 바뀌면서 사전학습 때 배운 언어 지식이 파괴될 수 있다. 이걸 파국적 망각(Catastrophic Forgetting)이라고 부른다. 뉴스 분류를 배우려다가 영어 자체를 잊어버리는 격이다. 작은 학습률은 기존 가중치를 아주 살짝만 조정한다. “영어를 이해하는 능력은 거의 그대로 두면서, 뉴스 카테고리를 구분하는 데 필요한 방향으로만 미세하게 틀어줘"라는 뜻이다. 이래서 “파인튜닝(fine-tuning, 미세 조정)“이라는 이름이 붙은 것이다.

                  num_train_epochs=1이라는 설정도 주목할 만하다. 단 한 에포크, 즉 12만 건의 데이터를 딱 한 바퀴만 돈다. DNA 분석에서 200 에포크를 돌았던 것과 대조적이다. BERT가 이미 강력한 언어 이해 능력을 갖추고 있기 때문에, 분류 작업에 적응하는 데는 한 바퀴만으로도 충분할 수 있다. 오히려 너무 많이 돌면 학습 데이터에 과적합되어 새로운 기사에 대한 일반화 능력이 떨어질 위험이 있다.

                  weight_decay=0.01은 L2 정규화다. 당뇨병성 망막증 모델에서 λ=0.1로 가중치가 커지는 걸 억제했던 것과 같은 원리인데, 여기서는 0.01로 더 약하게 건다. BERT의 가중치를 너무 강하게 제약하면 표현력이 줄어들 수 있기 때문이다.

                  #

                  #5 평가 지표

                  def compute_metrics(eval_pred):
    predictions = np.argmax(logits, axis=-1)
    precision, recall, f1, _ = precision_recall_fscore_support(
        labels, predictions, average="weighted"
    )
    acc = accuracy_score(labels, predictions)
    return {"accuracy": acc, "precision": precision, "recall": recall, "f1": f1}

                  이 프로젝트에서는 네 가지 지표를 동시에 측정한다. 정확도(Accuracy), 정밀도(Precision), 재현율(Recall), 그리고 F1 점수다.

                  정확도는 전체 중 맞춘 비율이다. 가장 직관적이지만, 이전에 봤듯이 불균형 데이터에서는 한계가 있다. 여기서는 데이터가 균등하니까 정확도가 그나마 의미 있지만, 더 세밀한 분석을 위해 다른 지표도 필요하다. 정밀도는 “모델이 경제 기사라고 예측한 것 중 실제로 경제 기사인 비율"이다. 모델이 자신 있게 말한 것이 얼마나 믿을 만한가를 측정한다. 재현율은 “실제 경제 기사 중 모델이 경제라고 잡아낸 비율"이다. 모델이 놓치는 게 얼마나 되는가를 측정한다.

                  이 둘 사이에는 긴장이 있다. 극단적으로 조심스러운 모델은 확실한 것만 “경제"라고 말하니까 정밀도가 높지만, 애매한 기사를 많이 놓쳐서 재현율이 낮다. 반대로 극단적으로 적극적인 모델은 조금이라도 경제 냄새가 나면 “경제"라고 말하니까 재현율은 높지만, 오판이 많아 정밀도가 낮다.

                  F1 점수는 이 둘의 조화평균으로, 정밀도와 재현율 사이의 균형을 하나의 숫자로 요약한다. 이 프로젝트에서 metric_for_best_model=“f1"로 설정한 건, 모델 선택의 기준으로 F1을 쓰겠다는 뜻이다. 정확도가 아니라 F1을 기준으로 삼는 건 더 균형 잡힌 모델을 선호하겠다는 의도다.

                  “weighted” 평균은 각 클래스의 샘플 수에 비례해서 가중 평균하는 방식이다. 여기서는 네 클래스가 균등하니까 단순 평균과 거의 같지만, 클래스 간 불균형이 있는 경우에도 공정한 평가를 보장하는 안전장치다.

                  지표계산식의미
                  Accuracy맞은 수 / 전체전체 정확도
                  PrecisionTP / (TP+FP)모델이 Business라 한 것 중 실제 Business 비율
                  RecallTP / (TP+FN)실제 Business 중 모델이 잡아낸 비율
                  F12·P·R / (P+R)Precision과 Recall의 조화평균

                  #6 MLflow: 실험 기록

                  with mlflow.start_run(run_name=f"bert-ag-news-ep{EPOCHS}-bs{BATCH_SIZE}") as run:
    mlflow.log_params({...})   # 하이퍼파라미터 기록
    trainer.train()
    mlflow.log_metrics({...})  # 성능 지표 기록
    mlflow.pytorch.log_model(model, "model")  # 모델 아티팩트 저장

                  MLflow는 실험 노트 같은 도구다. 어떤 하이퍼파라미터로 학습했고, 결과 지표가 어떻게 나왔고, 어떤 모델이 만들어졌는지를 자동으로 기록한다.

                  왜 이게 필요한가? 실제 프로젝트에서는 수십, 수백 번의 실험을 한다. 학습률을 바꿔보고, 배치 크기를 바꿔보고, 에포크 수를 바꿔보고. 이걸 수작업으로 엑셀에 기록하면 실수가 생기고 추적이 안 된다. MLflow가 매 실험마다 모든 설정과 결과를 자동으로 저장하고, 웹 인터페이스에서 실험들을 비교할 수 있게 해준다.

                  이건 이전 챕터들에는 없었던 요소다. 연구 코드에서는 print로 결과를 출력하고 넘어갔지만, 운영 시스템에서는 모든 실험의 이력이 추적 가능해야 한다. “지금 서비스 중인 모델은 어떤 설정으로 학습된 거지?” “지난달 실험한 모델과 비교하면 어떻게 다르지?“라는 질문에 답할 수 있어야 하기 때문이다.

                  #7 ONNX 변환: 서빙을 위한 최적화

                  torch.onnx.export(
    model,
    (dummy["input_ids"], dummy["attention_mask"]),
    ONNX_PATH,
    input_names=["input_ids", "attention_mask"],
    output_names=["logits"],
    dynamic_axes={
        "input_ids": {0: "batch_size", 1: "sequence"},  # 가변 크기 지원
        ...
    },
    opset_version=14,
)

                  모델이 학습을 마치면 바로 서비스에 투입하는 게 아니라 ONNX라는 형식으로 변환한다.

                  PyTorch는 학습에 최적화된 프레임워크다. 동적 그래프를 지원해서 코드를 유연하게 작성할 수 있고, 디버깅이 쉽다. 하지만 이 유연함에는 대가가 따른다. 추론(예측)할 때도 Python 인터프리터를 거치고, 매번 그래프를 동적으로 구성하니까 속도가 느리다.

                  ONNX는 모델의 연산 과정을 정적인 그래프로 고정시킨다. 학습할 때 필요했던 그래디언트 계산, 동적 분기 같은 것들을 다 제거하고, 순수하게 “입력 → 출력"의 연산만 남긴다. 이 정적 그래프 위에서 연산 융합(여러 연산을 하나로 합치기), 불필요한 연산 제거 같은 최적화를 적용할 수 있다.

                  dynamic_axes 설정은 배치 크기와 시퀀스 길이가 가변적임을 명시하는 것이다. 실제 서비스에서는 한 번에 하나의 요청이 올 수도 있고 여러 개가 묶여서 올 수도 있고, 문장 길이도 제각각이니까 이 유연성이 필요하다. 결과적으로 같은 모델, 같은 예측인데 ONNX가 PyTorch보다 수 배 빠르다. 하루에 수백만 건의 뉴스를 처리해야 하는 서비스에서 이 속도 차이는 서버 비용과 응답 시간에 직결되는 현실적인 문제다.

                  왜 ONNX가 빠른가?
- PyTorch: Python 오버헤드, 동적 그래프
- ONNX: 정적 그래프 최적화 (연산 융합, 불필요 연산 제거)

# 실제로 이 프로젝트에서 벤치마크한 결과: ONNX가 PyTorch보다 수 배 빠름.

                  #8 정리

                  12만 건의 균등 분포 뉴스 데이터를 준비하고, 사전학습된 BERT 위에 4클래스 분류 레이어를 얹고, 극도로 작은 학습률로 단 한 에포크를 파인튜닝하고, F1 점수로 성능을 평가하고, MLflow로 실험을 기록하고, ONNX로 서빙용 최적화를 한다.

                  여기서 핵심 교훈은, 사전학습된 거대 모델의 힘이다. 처음부터 모델을 만들었다면 12만 건으로는 부족했을 것이다. 하지만 이미 수십억 단어를 학습한 BERT를 출발점으로 삼으면, 12만 건의 파인튜닝만으로도 뛰어난 분류기를 만들 수 있다. 기존 지식 위에 새로운 기술을 쌓는 것, 그것이 파인튜닝의 본질이다.
