                    Genome Curation #3 Variant Calling

                    Genome Curation #3 Variant Calling #

                    #2026-04-02

                    #1

                    BAM 파일에는 각 read가 어느 염색체의 어느 위치에서 왔는지가 기록되어 있다. 이 맵핑된 read들을 참조 유전체와 비교해서 참조와 다른 부분 즉 변이(variant)를 찾아야하는데 이 과정을 변이 호출(variant calling) 또는 information calling이라고 한다.

                    reads with genome position (.bam)
        ↓  variant calling
information (.vcf or table)
  - SNV (단일 염기 변이)
  - Indel (짧은 삽입/결실)
  - CNV (복제 수 변이)
  - SV / Fusion (구조 변이 / 융합 유전자)

                    단순히 “참조와 다르면 변이"라고 판단할 수 없는게 시퀀싱 과정에서 발생한 오류도 참조와 다르게 보이기 때문이다. 예를 들어 어떤 위치를 30번 읽었는데 그중 1번만 참조와 다르다면 시퀀싱 오류일 가능성이 높고, 15번이 다르다면 진짜 변이(이형접합, heterozygous)일 가능성이 높다. 변이 호출 알고리즘은 이런 통계적 판단을 수행한다.

                    #

                    #2 변이 유형별 전문 도구들 (Callers)

                    변이의 종류마다 나타나는 패턴이 완전히 다르기 때문에 각 유형을 전문적으로 탐지하는 별도의 도구(caller)가 존재한다.

                    #

                    SNV (단일 염기 변이) 전용 도구

                    SNV는 DNA 문자 하나가 다른 문자로 바뀐 것이다. 참조 유전체에서 특정 위치가 A인데, 그 위치 read들 중 많은 read가 G로 읽혔다면 SNV 후보가 된다.

                    SNV를 전문적으로 찾는 도구에는 Mutect, Strelka, VarScan2, UnifiedGenotyper, SomaticSniper, SMuFin, CaVEMan, EBCall 등이 있다. 이 도구들은 각각 다른 통계 모델을 사용하지만 공통적으로 해당 위치를 덮는 읽기들의 염기 분포, 품질 점수, 맵핑 품질 등을 종합적으로 고려해서 “이것이 진짜 변이인가 시퀀싱 오류인가"를 판단한다.

                    특히 체세포 돌연변이(somatic mutation) 를 찾을 때는 종양(tumor) 시료와 정상(normal) 시료를 함께 분석한다. 종양에서만 나타나고 정상에서는 나타나지 않는 변이가 체세포 돌연변이이기 때문이다. Mutect, SomaticSniper 같은 도구가 이런 종양-정상 비교 분석에 특화되어 있다.

                    #

                    Short Indel (짧은 삽입/결실) 전용 도구

                    Indel은 짧은 염기가 삽입되거나 결실된 것이다. SNV보다 탐지하기 어려운데, 삽입이나 결실이 있으면 그 주변의 읽기 정렬이 흐트러지기 때문이다. 참조 유전체에 비해 3bp가 삽입된 읽기는 해당 위치에서 정렬이 밀리게 되고, 이것을 정확히 인식하려면 단순한 염기 비교가 아니라 정렬 자체를 재조정해야 한다.

                    Strelka, Pindel, Dindel, Indelocator, SOAPindel 등이 indel 탐지에 쓰인다. Pindel은 특히 split-read 분석에 강한데, 하나의 읽기가 참조 유전체에 두 조각으로 나뉘어 맵핑되는 패턴을 이용해 삽입/결실 위치를 정밀하게 찾아낸다.

                    #

                    SV (구조 변이) 전용 도구

                    구조 변이는 100bp 이상의 큰 규모 변이로, 전좌(translocation), 역위(inversion), 큰 결실, 큰 삽입 등을 포함한다. 구조 변이를 탐지하는 방법은 SNV와 근본적으로 다르다.

                    핵심 단서는 비정상적으로 맵핑된 읽기 쌍(discordant read pairs) 이다. paired-end 시퀀싱에서 R1과 R2는 같은 DNA 조각에서 나왔으므로 참조 유전체 위에서 300~500bp 정도 떨어져 서로 마주보는 방향으로 맵핑되어야 한다. 그런데 R1은 7번 염색체에, R2는 14번 염색체에 맵핑되었다면? 이것은 7번과 14번 염색체 사이에 전좌가 일어났다는 단서다. 또는 R1과 R2가 같은 염색체에 맵핑되었지만 예상보다 수만 bp 떨어져 있다면 큰 결실의 증거가 된다.

                    dRanger, BreakDancer, BreakPointer, GRAFT, DELLY, Breakmer, BRASS 등이 이런 비정상적 맵핑 패턴을 분석하여 구조 변이를 찾아낸다. 각 도구는 discordant read pairs, split reads, 읽기 깊이 변화 등 서로 다른 증거 유형에 가중치를 다르게 두고 있다.

                    #

                    Copy Number (복제 수 변이) 전용 도구

                    복제 수 변이(CNV/CNA)는 DNA의 특정 구간이 정상보다 많이 복제되거나 삭제된 것이다. 탐지 원리는 유전체의 특정 구간을 덮는 읽기의 개수, 즉 읽기 깊이(read depth) 를 관찰한다. 정상적으로 2개의 복사본이 있는 구간은 평균적인 읽기 깊이를 보인다. 그런데 특정 구간의 읽기 깊이가 평균의 2배라면 4개의 복사본(duplication)이 있을 가능성이 높고, 읽기 깊이가 절반이라면 1개의 복사본(hemizygous deletion)만 남았을 가능성이 높으며, 읽기가 전혀 없다면 양쪽 복사본 모두 삭제(homozygous deletion)되었을 수 있다.

                    ABSOLUTE, PICNIC, HMMCOPY, DNAcopy, VarScan2, QDNAseq, CNVkit 등이 이 분석을 수행한다. 단순히 읽기 깊이만 보는 것이 아니라, GC 함량에 따른 시퀀싱 편향, 맵핑 가능성(mappability), 종양 순도(tumor purity), 이형접합도 소실(LOH) 등 다양한 요인을 보정해야 정확한 복제 수를 추정할 수 있다.

                    #

                    #3 Mutect2를 이용한 체세포 변이 호출 파이프라인

                    다양한 caller 중에서 Mutect2는 현재 체세포 SNV와 short indel을 찾는 데 가장 널리 사용되는 도구다. Mutect2는 GATK(Genome Analysis Toolkit) 라는 대규모 유전체 분석 프레임워크에 포함되어 있다. GATK는 Broad Institute에서 개발한 것으로, 유전체 분석의 사실상 표준 도구 모음이다.

                    #

                    Mutect2 파이프라인은 크게 세 단계로 구성된다.

                    1단계: Mutect2 실행 (변이 후보 생성)

                    bashgatk Mutect2 \
  -R grch37.fa \
  -I HCC1.RGadded.marked.bam \
  -O HCC1.mutect2.vcf
                    • gatk Mutect2
                      • GATK 프레임워크에서 Mutect2 알고리즘을 호출
                    • -R grch37.fa
                      • 참조 유전체 파일을 지정. read mapping에서 사용한 것과 동일한 참조 유전체여야 한다. grch37.fa는 GRCh37(hg19) 버전의 인간 참조 유전체.
                    • -I HCC1.RGadded.marked.bam
                      • 입력 BAM 파일. 파일 이름에서 RGadded는 읽기 그룹(Read Group) 정보가 추가되었다는 뜻이고, marked는 중복 읽기(duplicate reads)가 표시되었다는 뜻.
                      • 중복 읽기(duplicate reads)?
                        • 라이브러리 제작 과정에서 PCR 증폭을 하면 같은 DNA 조각이 여러 개로 복제된다. 이 복제물들은 유전체의 같은 위치에 맵핑되지만, 독립적인 증거가 아니다. 같은 원본의 복사물이기 때문이다. 만약 중복 읽기를 독립적인 증거로 취급하면 변이 호출의 통계가 왜곡된다. 따라서 MarkDuplicates라는 도구로 중복 읽기를 미리 표시해두고, Mutect2가 이를 무시하도록 한다.
                    • -O HCC1.mutect2.vcf
                      • 출력 VCF 파일.

                    위 예시는 tumor only mode, 즉 종양 시료만으로 변이를 호출하는 경우다. 정상 시료가 함께 있는 일반적인 경우라면 다음과 같이 두 개의 BAM 파일을 입력하고 어느 것이 정상인지 지정한다.

                    bashgatk Mutect2 \
  -R human_reference.fa \
  -I tumor.bam \
  -I normal.bam \
  -normal normal_sample_name \
  -O output.vcf
                    • -I tumor.bam 과 -I normal.bam
                      • 두 BAM 파일을 모두 입력
                      • -normal normal_sample_name 으로 어느 샘플이 정상인지 지정. Mutect2는 종양에서는 관찰되지만 정상에서는 관찰되지 않는 변이를 체세포 돌연변이로 판단한다.

                    Mutect2의 내부 알고리즘은 베이지안 통계 모델에 기반한다. 각 위치에서 “이 위치에 변이가 있을 확률"을 사전 확률(prior)과 관찰된 읽기 데이터(likelihood)로부터 사후 확률(posterior)로 계산한다. 이 계산에서 읽기의 품질 점수, 맵핑 품질, 해당 위치의 읽기 깊이, 대립 유전자 빈도(allele frequency) 등이 모두 고려된다.

                    #

                    2단계: FilterMutectCalls (필터링)

                    bashgatk FilterMutectCalls \
  -V HCC1.mutect2.vcf \
  --max-alt-allele-count 5 \
  -O HCC1.mutect2.filter.vcf

                    생성된 VCF 파일에는 Mutect2가 후보로 잡은 모든 변이가 들어있다. 이 중에는 진짜 변이도 있지만, 다양한 인공 오류(artifact)도 포함되어 있다. FilterMutectCalls는 여러 기준으로 후보들을 걸러낸다.

                    • -V HCC1.mutect2.vcf
                      • 필터링할 입력 VCF 파일.
                    • –max-alt-allele-count 5
                      • 필터링 옵션 중 하나로, 한 위치에서 대체 대립유전자(alt allele)의 수가 5개를 초과하면 걸러냄. 정상적인 체세포 돌연변이는 보통 하나의 대체 대립유전자만 가지므로, 대체 대립유전자가 너무 많으면 시퀀싱 오류나 맵핑 문제일 가능성이 높다.
                    • -O HCC1.mutect2.filter.vcf
                      • 필터링 결과가 저장되는 출력 VCF 파일.

                    FilterMutectCalls는 변이를 삭제하는 것이 아니라, VCF 파일의 FILTER 열에 통과 여부를 기록한다. 필터를 통과한 변이는 PASS로, 통과하지 못한 변이는 실패 이유(예: weak_evidence, strand_bias 등)가 기록된다.

                    #

                    3단계: Annotation (변이 주석)

                    필터링을 통과한 변이들에 대해 “이 변이가 어떤 유전자에 있고, 어떤 종류의 변화를 일으키는지"를 주석을 단다. 대표적인 도구가 ANNOVAR와 VEP(Ensembl Variant Effect Predictor) 다.

                    이 도구들은 각 변이에 대해 다음과 같은 정보를 추가한다. 해당 변이가 어느 유전자의 어느 엑손에 위치하는지, 코딩 변이라면 과오(missense)인지 무의미(nonsense)인지 침묵(silent)인지, 단백질 수준에서 어떤 아미노산 변화가 일어나는지(예: p.V600E는 600번째 아미노산이 발린에서 글루탐산으로 바뀜), 그리고 이미 알려진 변이인지(dbSNP, COSMIC 등의 데이터베이스 조회) 등을 분석하여 VCF 파일에 추가한다.

                    입력:  HCC1.mutect2.filter.vcf  (필터링 완료된 변이 목록)

출력:  annotated VCF 또는 테이블
  - 각 변이의 유전자, 위치, 효과, 데이터베이스 매칭 결과 등

                    #

                    #4 VCF 파일 형식: 변이 정보의 표준 포맷

                    VCF(Variant Call Format) 는 유전체 변이 정보를 저장하는 표준 파일 형식이다. VCF 파일은 크게 헤더(header) 와 본문(body) 으로 나뉜다.

                    #

                    VCF 헤더

                    헤더는 ##으로 시작하는 줄들로, 파일의 메타데이터를 담는다. 좋은 VCF 파일과 나쁜 VCF 파일의 차이는 헤더에서 드러난다.

                    좋은 VCF 헤더의 예: 
- ##reference에 어떤 참조 유전체 서열을 사용했는지 명시되어 있고, ##source에 어떤 변이 호출 도구(예: Mutect2)를 사용했는지 기록되어 있으며, FILTER, INFO, FORMAT 필드의 정의가 모두 설명되어 있다. 
- 나중에 이 VCF 파일을 받은 사람이 "이 파일이 어떻게 만들어졌는지"를 헤더만 보고 완전히 이해할 수 있다.

나쁜 VCF 헤더의 예: 
- 참조 유전체가 무엇인지 기록되어 있지 않고, 어떤 도구로 호출했는지 알 수 없으며, FILTER 열에 LowFreq 같은 값이 나오는데 그것이 정확히 어떤 기준으로 판정된 것인지 정의가 없다. 
- 이런 VCF 파일은 재현 불가능하고, 다른 분석자가 해석하기 어렵다.

                    VCF 헤더는 분석의 재현성(reproducibility) 을 위해 반드시 충실하게 작성해야 한다.

                    #

                    VCF 본문 (Body)

                    본문은 탭으로 구분된 열(column)들로 구성된다. 각 행이 하나의 변이를 나타낸다.

                    #CHROM  POS       ID    REF  ALT  QUAL    FILTER  INFO           FORMAT    SAMPLE1  SAMPLE2
chr7    55249071  .     C    T    5000    PASS    DP=120;AF=0.45 GT:AD:DP  0/1:55,65:120  ...
                    • CHROM
                      • 변이가 발견된 염색체 번호. 위 예에서는 7번 염색체다. POS는 참조 유전체 위에서의 위치(좌표)다. 55249071이면 7번 염색체의 55,249,071번째 염기 위치다.
                    • ID
                      • 해당 변이에 부여된 식별자. dbSNP에 등록된 변이라면 rs번호(예: rs1234567)가 들어가고, 알려지지 않은 변이라면 .(마침표)로 표시된다.
                    • REF, ALT
                      • REF는 참조 유전체의 염기. 위 예에서는 C다. ALT는 실제 관찰된 대체 염기(alternative allele). 위 예에서는 T이므로 이 위치에서 C→T 치환이 일어난 것이다.
                    • QUAL, FILTER
                      • QUAL은 이 변이 호출의 품질 점수. Phred 스케일로 표현되며, 높을수록 신뢰도가 높다. 5000이면 매우 확실한 변이라는 뜻. FILTER는 필터링 통과 여부. PASS면 모든 필터를 통과한 것이고 그렇지 않으면 실패한 필터의 이름이 기록된다(예: strand_bias, weak_evidence).
                    • INFO
                      • 변이에 대한 추가 정보. DP=120은 해당 위치의 총 읽기 깊이가 120이라는 뜻이고 AF=0.45는 대체 대립유전자 빈도(allele frequency)가 45%라는 뜻.
                    • FORMAT
                      • 뒤에 이어지는 샘플별 데이터의 형식. GT:AD:DP이면 유전형(Genotype), 대립유전자별 읽기 깊이(Allelic Depth), 총 읽기 깊이(Depth) 순서로 기록된다는 뜻이다.
                    • SAMPLE1 (SAMPLE2, …)
                      • 각 샘플의 실제 데이터. 0/1:55,65:120이면 유전형이 이형접합(0/1, 참조 대립유전자 하나 + 대체 대립유전자 하나), 참조 염기를 지지하는 읽기가 55개, 대체 염기를 지지하는 읽기가 65개, 총 읽기 깊이가 120이라는 뜻이다.

                    #

                    VCF 해석 시 주의사항

                    FILTER=PASS라도 거짓 양성일 수 있다. FILTER 열에 PASS라고 기록되어 있어도 그것이 100% 진짜 변이라는 보장은 없다. 거짓 양성(false positive) 의 가능성은 항상 존재한다. 대표적인 원인으로는 참조 유전체의 반복 서열 영역에서 발생하는 맵핑 오류, PCR 증폭 편향으로 인해 특정 대립유전자가 과대 표현되는 경우, 또는 유전체의 상동 영역(paralogous region)으로 인한 잘못된 맵핑 등이 있다.

                    따라서 중요한 변이를 발견했을 때는 반드시 BAM 파일을 직접 확인해야 한다. IGV(Integrative Genomics Viewer) 같은 유전체 브라우저로 해당 위치의 읽기들을 직접 눈으로 보면, 변이를 지지하는 읽기들이 고르게 분포되어 있는지, 한쪽 가닥에만 편중되어 있지는 않은지(strand bias), 읽기의 끝부분에만 변이가 나타나지는 않는지(position bias) 등을 확인할 수 있다. 컴퓨터 알고리즘이 놓치는 미묘한 패턴을 사람의 눈이 잡아내는 경우가 적지 않다.

                    #

                    Tumor only mode의 한계

                    앞서 Mutect2 예시에서 본 것처럼 종양 시료만으로 변이를 호출하는 tumor only mode가 있다. 이 모드는 정상 시료가 없어서 종양-정상 비교가 불가능하다. 그 결과 발생하는 근본적인 문제가 있다. 발견된 변이가 체세포 돌연변이(somatic mutation, 암에서 새로 생긴 변이) 인지 생식세포 변이(germline variant, 원래부터 가지고 있던 변이) 인지 구분할 수 없다는 것이다.

                    예를 들어 EGFR 유전자에서 변이가 발견되었다고 하자. 이것이 암세포에서 새로 발생한 체세포 돌연변이라면 치료 표적이 될 수 있지만, 원래부터 환자가 가지고 있던 생식세포 다형성이라면 치료적 의미가 완전히 다르다. 정상 시료 없이는 이 구분이 원천적으로 불가능하다. gnomAD 같은 인구 집단 데이터베이스에서 흔한 생식세포 변이를 걸러내는 방법으로 부분적으로 보완할 수 있지만, 완벽하지 않다.

                    #

                    #5 전체 파이프라인 흐름

                    [BAM 파일]
 HCC1.RGadded.marked.bam   (읽기 그룹 추가 + 중복 읽기 표시 완료)

         ↓ 1단계: gatk Mutect2
           + 참조 유전체 (grch37.fa)
           + [선택] 정상 시료 BAM

[Raw VCF]
 HCC1.mutect2.vcf           (모든 변이 후보, 필터링 전)

         ↓ 2단계: gatk FilterMutectCalls
           + 다양한 필터링 기준 적용

[Filtered VCF]
 HCC1.mutect2.filter.vcf    (FILTER 열에 PASS / 실패 사유 기록)

         ↓ 3단계: Annotation (ANNOVAR 또는 VEP)
           + 유전자명, 변이 유형, 아미노산 변화
           + 데이터베이스 매칭 (dbSNP, COSMIC 등)

[Annotated VCF / 테이블]
 최종 결과: 각 변이의 위치, 종류, 임상적 의미

         ↓ 수동 검증
           + IGV로 BAM 파일 직접 확인
           + 거짓 양성 여부 판단

                    #

                    #6 핵심 요약

                    • 변이 호출(variant calling) 은 BAM 파일의 맵핑된 읽기를 참조 유전체와 비교하여 변이를 통계적으로 판별하는 과정이다.
                    • 변이 유형별로 전문 도구가 존재한다. SNV(Mutect 등), Indel(Pindel 등), SV(DELLY 등), CNV(CNVkit 등)는 각각 다른 알고리즘과 증거 유형을 사용한다.
                    • Mutect2 파이프라인은 변이 호출 → 필터링(FilterMutectCalls) → 주석(ANNOVAR/VEP)의 세 단계로 구성된다.
                    • VCF 파일은 변이 정보의 표준 포맷으로, 충실한 헤더 작성이 재현성의 핵심이다. 본문의 CHROM, POS, REF, ALT, QUAL, FILTER, INFO, FORMAT, SAMPLE 열이 각 변이의 모든 정보를 담는다.
                    • FILTER=PASS라도 반드시 BAM 파일에서 수동 검증해야 하며, tumor only mode에서는 체세포/생식세포 변이 구분이 불가능하다는 한계를 인식해야 한다.
